# Clarity — Agent Guide with code-review-graph

## Project Context

Clarity is a Flutter weather app built with **Clean Architecture** (feature-first),
**BLoC** for state management, **Firebase** for auth/messaging/crashlytics,
**dartz** for functional error handling, and **GetIt** for dependency injection.

The project has a **knowledge graph** (code-review-graph) that indexes every
Dart file into a structural map of functions, classes, imports, calls, and
inheritance. Use it before reaching for Grep/Glob/Read.

## Architecture at a Glance

```
lib/
├── core/               ← shared infra (DI, theme, errors, services, router, config)
│   ├── config/         ← EnvConfig, env loading
│   ├── di/             ← injection_container.dart (GetIt sl)
│   ├── error/          ← Failure + Exception types
│   ├── responsive/     ← Breakpoints (phone <600, tablet <1024, desktop)
│   ├── router/         ← GoRouter config
│   ├── services/       ← LocationService, NotificationService
│   ├── theme/          ← AppColors, AppTheme (Bricolage Grotesque, neumorphic)
│   └── utils/          ← Localizer, weather_icon_mapper
├── features/
│   ├── auth/           ← Firebase Auth (email, Google, anonymous)
│   │   ├── data/       ← FirebaseAuthDataSource, AuthUserModel, AuthRepositoryImpl
│   │   ├── domain/     ← AuthUser entity, AuthRepository interface
│   │   └── presentation/ ← AuthBloc, LoginPage, ProfilePage, AuthGate
│   ├── navigation/     ← MainScreen (page nav), MenuScreen (drawer)
│   ├── settings/       ← SettingsBloc, SettingsPage, AppSettings entity
│   └── weather/        ← OpenWeather API
│       ├── data/       ← WeatherRemoteDataSource, WeatherModel, WeatherRepositoryImpl
│       ├── domain/     ← Weather entity, WeatherRepository, GetWeather UseCase
│       └── presentation/ ← WeatherBloc, WeatherPage, ForecastPage, ClayContainer
└── firebase_options.dart ← generated by flutterfire configure
```

**Layer flow:** `presentation → domain ← data`
- Domain is pure Dart: entities, repository interfaces, use cases. No Flutter/dio/bloc.
- Data implements domain interfaces: remote data sources, models, repository impls.
- Presentation: BLoC (events/states), pages, widgets.
- Error handling: `Either<Failure, T>` everywhere; never throw across layers.
- DI: `sl` (GetIt) in `injection_container.dart`. BLoCs = `registerFactory`; repos/datasources = `registerLazySingleton`.

## code-review-graph — How It Works

The graph is built on **Tree-sitter** (parses Dart ASTs) and stored in SQLite at
`.code-review-graph/graph.db`. It tracks:

| Node kind | What it captures |
|-----------|-----------------|
| `File` | Every `.dart`, `.swift`, `.kt`, `.js` file |
| `Class` | BLoCs, entities, repositories, data sources, widgets, models |
| `Function` | Methods, top-level functions, constructors |

| Edge kind | What it captures |
|-----------|-----------------|
| `CALLS` | Function/method calls (e.g. `WeatherBloc._onGetWeather` calls `getWeather`) |
| `IMPORTS_FROM` | Import statements (relative + some `package:` within project) |
| `INHERITS` | `extends`/`implements`/`with` (e.g. `AuthRepositoryImpl implements AuthRepository`) |
| `CONTAINS` | File contains class/function |
| `REFERENCES` | Type annotations, variable references |

**Current stats:** 391 nodes, 1953 edges, 82 files, 10 communities.

### Known Dart Limitations — VERIFIED 2026-07-21

> These were tested empirically against this exact graph. Two of the three
> previously-documented claims here were **wrong**; this is the corrected version.

1. **`callees_of` on a Dart class or method ALWAYS returns 0.**
   Dart `CALLS` edges use the **file** as `source_qualified` — they never carry a
   `::Class.method` suffix, so no class/function node is ever a CALLS *source*.
   ❌ `callees_of("WeatherBloc")` → 0
   ❌ `callees_of("WeatherBloc._fetchWeatherByCity")` → 0
   ✅ `callees_of("lib/features/weather/presentation/bloc/weather_bloc.dart")` → 15 results
   **Query the file path.** Do *not* fall back to Grep — the data is there.

2. **Use QUALIFIED names for `callers_of` / `callees_of`, not bare names.**
   (The previous version of this doc said the opposite.) A bare name matches the class
   *and* all its methods via FTS and returns `status: "ambiguous"` with candidates.
   ❌ `callers_of("WeatherBloc")` → ambiguous (5 candidates)
   ✅ `callers_of("/abs/path/…/weather_bloc.dart::WeatherBloc")` → 1 result (injection_container.dart)
   Get the qualified name from `semantic_search_nodes_tool` first.

3. **Only ~24% of CALLS targets resolve to real nodes** (Dart: 27%). The rest are bare
   names like `WeatherError`, `emit`, `fold`. Absence of an edge is *not* proof of no call.

4. **`package:clarity/...` resolution is moot** — this codebase has **zero** such imports;
   it uses relative imports exclusively. Relative imports resolve correctly to absolute
   paths. External `package:flutter/...` stay as unresolved literal strings (expected).

5. **Communities are directory-based, not feature-based.** There is ONE `lib/features`
   community (182 nodes, misleadingly named `pages-weather`) containing auth + weather +
   settings + navigation, and one `lib/core`. Do not expect per-feature clusters.

6. **"0 cross-community edges" is an artifact, not evidence of decoupling.** All 82 `File`
   nodes have `community_id = NULL`, and `IMPORTS_FROM` edges are File→File — so no edge
   ever has two community-assigned endpoints. Coupling warnings cannot fire.

7. **Hub nodes are pages, not the DI container**, because CALLS edges are file-sourced and
   large UI files accumulate the most edges. `injection_container.dart` ranks ~9th.

### If the MCP tools aren't loaded

They only initialise when a session starts *after* `.mcp.json` exists. If they're missing,
**don't fall back to Grep** — the CLI has an exact equivalent for every tool and reads the
same `graph.db`:

```bash
code-review-graph search "WeatherBloc" --kind Class     # semantic_search_nodes_tool
code-review-graph query callers_of "<qualified_name>"   # query_graph_tool
code-review-graph impact --files <path>                 # get_impact_radius_tool
code-review-graph communities / architecture            # list_communities / overview
code-review-graph detect-changes                        # detect_changes_tool
code-review-graph dead-code / large-functions
```

Note the generated skills in `.claude/skills/` reference a few tool names that may not
exist (`get_minimal_context`, `list_graph_stats`, `list_flows`). If a call errors with an
unknown tool, use the equivalent from the table above rather than retrying.

## Query Recipes for Flutter + Clean Architecture

### "What calls this BLoC method?"

```json
query_graph_tool(pattern="callers_of", target="WeatherBloc._onGetWeather")
```

### "What does this UseCase depend on?"

```json
query_graph_tool(pattern="callees_of", target="GetWeather")
```

### "Trace the full dependency chain from a page"

```json
traverse_graph_tool(start_node="WeatherPage", direction="both", max_depth=3, max_tokens=2000)
```

### "What is the blast radius of changing this entity?"

```json
get_impact_radius_tool(changed_files=["lib/features/weather/domain/entities/weather.dart"])
```

### "Review my current changes with risk scoring"

```json
detect_changes_tool()
```

### "Show me the architecture"

```json
get_architecture_overview_tool()
```

### "Find all BLoCs in the project"

```json
semantic_search_nodes_tool(query="Bloc", node_type="Class", language="dart")
```

### "Find all UseCases"

```json
semantic_search_nodes_tool(query="UseCase", node_type="Function", language="dart")
```

### "What tests exist for the auth feature?"

```json
query_graph_tool(pattern="tests_for", target="AuthRepository")
```

### "Find the most-connected classes (DI hubs)"

```json
get_hub_nodes_tool(top_n=10)
```

## Workflow for Agents

1. **Session start** — Check `code-review-graph status` to confirm graph is fresh.
2. **Exploring code** — Use `semantic_search_nodes_tool` or `query_graph_tool`
   BEFORE Grep. The graph gives you structural context (callers, dependents,
   test coverage) that file scanning cannot.
3. **Making changes** — After editing, use `detect_changes_tool` or
   `get_impact_radius_tool` to understand blast radius before committing.
4. **Code review** — Use `get_review_context_tool` for token-optimized snippets.
5. **Architecture questions** — Use `get_architecture_overview_tool` and
   `list_communities_tool`.
6. **Fallback** — If the graph returns 0 results for something you know exists,
   fall back to Grep/Glob/Read. Dart package: imports may not resolve.
